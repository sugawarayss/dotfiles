# ADR(アーキテクチャ意思決定記録) Format

ADRは `docs/adr/` ディレクトリ内に配置し、`0001-slug.md`、`0002-slug.md` のように連番を使用する。

`docs/adr` ディレクトリは遅延生成(最初のADRが必要になった時点で初めて作成)すること。

## テンプレート

```markdown
# {決定事項の簡潔なタイトル}

{1-3 文章: 背景、何を決定したか、およびその理由}
```

ADRは1段落のみでも良い。
価値があるのは「決定されたこと」と「その理由」を記録することであって、セクションを埋めることではない。

## 任意事項

真に価値がある場合のみ含めること。大半のADRでは不要である。

- ステータスのフロントマター(proposed（提案中） | accepted（承認済） | deprecated（非推奨） | superseded by ADR-NNNN（ADR-NNNNにより置き換え）): 意思決定を再検討する際に有用。
- 検討した選択肢（Considered Options）: 却下された代替案に、あえて記憶しておく価値がある場合のみ。
- 影響・結果（Consequences）: 一見して分からないような下流への影響を、明記しておく必要がある場合のみ。

## 採番

`docs/adr/` 内をスキャンして既存の最大番号を確認し、それに「1」を加算（インクリメント）する。

## ADRを提案すべきタイミング

以下の3つの条件が **すべて** 満されている必要がある。

1. 覆すのが難しい: 後から考え直したときの変更コストが看過できない。
2. 文脈がないと不可解である: 将来コードを見た人が「一体なぜこんな書き方にしたんだ？」と疑問に思う。
3. 現実的なトレードオフの結果である: 他の選択肢が存在し、特定の理由からそのうちの1つを選んだ。

簡単に覆せる決定なら、ADRは不要である（ただ元に戻せばよいため）。
不可解でなければ、誰も理由を疑問に思わない。
本当の代替案がなかったのであれば、「当然のことをした」以外に記録すべきことはない。

### ADRの対象となるもの

- **アーキテクチャの構造**: 「モノレポを採用する」「書き込みモデルはイベントソーシングとし、読み込みモデルはPostgreSQLにプロジェクション（投影）する」
- **コンテキスト間の統合パターン**: 「注文管理と請求管理は、同期的なHTTPではなく、ドメインイベントを介して通信する」
- **ロックインを伴う技術選定**: データベース、メッセージバス、認証プロバイダ、デプロイ先など。すべてのライブラリではなく、入れ替えに数ヶ月（1クォーター）かかるものに限定。
- **境界とスコープの決定**: 「顧客データは顧客コンテキストが所有し、他のコンテキストはIDでのみ参照する」。「明示的に『やらない』と決めたこと」は、「やること」と同じくらい価値がある。
- **あえて定石から外れた決定**: 「Xという理由から、ORMではなく手動のSQLを使用する」など。一般的な読者が逆の想定をしそうなすべての事柄。これにより、次のエンジニアが「意図的な設計」をバグと勘違いして「修正」してしまうのを防ぐ。
- **コードからは見えない制約**: 「コンプライアンス要件のため、AWSは使用できない」「パートナー企業のAPI契約のため、レスポンスタイムは200ms未満でなければならない」
- **却下した理由が分かりにくい代替案**: GraphQLを検討した上で、微妙な（ニッチな）理由からRESTを選んだのであれば、それを記録しておく。そうしなければ、半年後に誰かがまたGraphQLの導入を提案することになる。
