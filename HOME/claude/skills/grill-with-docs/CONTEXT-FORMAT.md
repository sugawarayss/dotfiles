# CONTEXT.md Format

## 構造

```markdown
# {Context 名}

{このContextがどのようなもので、なぜ存在するのかを簡潔に説明する文章}

## 用語


**Order**:
{用語の明確な説明文}
_Avoid_: Purchase, transaction

**Invoice**:
配送後に顧客に送付する支払情報
_Avoid_: Bill, payment request

**Customer**:
注文を行う個人または組織。
_Avoid_: Client, buyer, account

## 用語間の連携

- 1つの **Order** は 1つ以上の **Invoice**と紐付きます
- 1つの **Invoice** は 確実に1つの **Customer** に紐付きます

## 確認の例

> **Dev:** "**Customer** が **Order** を入れた時、我々は即時に **Invoice** を作成しますか?"
> **Domain expert:** "いいえ。 **Invoice** は **Fulfillment** が確認された時に1度だけ作成します。"

## 曖昧な用語

- 「account」は、**Customer** と **User** の両方を意味するために使用されていましたが、解決済みです。これらは異なる概念です。
```

## ルール

- **明確な意見を持つ** 同じ概念を表す言葉が複数存在する場合は、最適な言葉を選び、他の言葉はエイリアスとして列挙して使用を避ける。
- **曖昧な用語は明示的に指摘する** 用語が曖昧に使用されている場合は、「曖昧な用語」欄に明確な解決策を記載する。
- **定義は簡潔にする** 定義は最大1文にとどめる。「何をするか?」ではなく、「何であるか?」を定義する。
- **関係性を示す** 用語名は太字にし、明確な箇所にはカーディナリティ（対応関係／要素数）を明示すること。
- **このプロジェクトのコンテキストに固有の用語のみを記載する** 一般的なプログラミング概念（タイムアウト、エラーの種類、ユーティリティパターンなど）は、プロジェクトで頻繁に使用される場合でも含めない。用語を追加する前に、「これはこのコンテキストに固有の概念か、それとも一般的なプログラミング概念か？」と自問し、前者のみを含める。
- **自然なまとまりができる場合は、小見出しの下に用語をグループ化する** すべての用語が1つのまとまった分野に属している場合は、階層のない箇条書き（フラットリスト）で構わない。
- **対話例を記載する** 開発者（デベロッパー）とドメイン専門家（業務精通者）の会話を通じて、用語がどのように自然に関連し合っているかを示し、類似した概念同士の境界線を明確にすること。

## 単一Context vs 複数Context のリポジトリ

**単一Contextのリポジトリ(大半のリポジトリ)**: リポジトリのルートに `CONTEXT.md` を1つだけ配置する。

**複数Contextのリポジトリ**: リポジトリのルートにある CONTEXT-MAP.md に、各コンテキストの一覧、配置場所、およびそれらの相互関係を記述する。

```md
# Context Map

## Contexts

- [Ordering](./src/ordering/CONTEXT.md) — 受領した または 追跡中の顧客の注文情報
- [Billing](./src/billing/CONTEXT.md) — 生成した請求情報 と 処理中の支払情報
- [Fulfillment](./src/fulfillment/CONTEXT.md) — 倉庫在庫の管理と発送情報

## Relationships

- **Ordering → Fulfillment**: Ordering は `OrderPlaced` イベントを発行する; Fulfillment がそれに伴って在庫のピッキングを開始する
- **Fulfillment → Billing**: Fulfillment は `ShipmentDispatched` イベントを発行する; Billing がそれに伴って請求情報を生成する
- **Ordering ↔ Billing**: Shared types for `CustomerId` と `Money` を共有する
```

このスキルは、どちらの構造が適用されるかを自動的に推論する:

- `CONTEXT-MAP.md` が存在する場合: それを読み込んでコンテキストを特定する
- ルートの `CONTEXT.md` のみが存在する場合: 単一コンテキストとして扱う
- どちらも存在しない場合: 最初の用語（term）が解決された時点で、ルートの `CONTEXT.md` を遅延生成（オンデマンドで作成）する

複数コンテキストが存在する場合、現在のトピックがどのコンテキストに関連しているかを推論すること。判断がつかない場合は、ユーザーに確認すること。
