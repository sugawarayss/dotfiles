# Domain Modeling

設計を進める中で、プロジェクトのドメインモデルを積極的に構築し、磨き上げていきましょう。これが「能動的な」規律です。
つまり、用語を検証し、エッジケースのシナリオを考案し、用語集や決定事項が明確になった瞬間に書き留めるということです。
（単に語彙を確認するために`CONTEXT.md`を「読む」だけではこのスキルとは言えません。それはどんなスキルでもできる一行の習慣です。このスキルは、モデルをただ利用するだけでなく、変更を加える際に必要となるものです。）

## ディレクトリ構造

殆どのリポジトリは1つのコンテキストを持ちます:

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

プロジェクトルートに`CONTEXT-MAP.md`が存在する場合は複数のコンテキストを持ちます。
マップはそれぞれのコンテキストの場所を示しています。

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← system-wide decisions
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← context-specific decisions
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

ファイルは遅延作成します。つまり、書き込む内容がある場合にのみ作成します。
`CONTEXT.md` が存在しない場合は、最初の用語が解決されたときに作成します。
`docs/adr/` が存在しない場合は、最初の ADR が必要になったときに作成します。

## セッション中

### 用語集との照合

ユーザーが`CONTEXT.md`に記載されている用語と矛盾する用語を使用した場合は、すぐに指摘してください。
「用語集では『キャンセル』はXと定義されていますが、あなたはYを意味しているようです。どちらが正しいですか？」

### 曖昧な言葉を明確にする

ユーザーが曖昧な用語や多義的な用語を使用している場合は、より正確な標準用語を提案してください。
「『アカウント』とおっしゃっていますが、それは顧客のことですか、それともユーザーのことですか？それらは異なるものです。」

### 具体的なシナリオについて話し合う

ドメイン間の関係性を議論する際は、具体的なシナリオを用いてストレステストを実施しましょう。
エッジケースを検証し、概念間の境界を明確にするようユーザーに促すようなシナリオを考案してください。

### コードとの整合性を確認する

ユーザーが操作方法を説明した際は、コードがそれと一致しているか確認してください。
矛盾が見つかった場合は、それを指摘してください。
「あなたのコードは注文全体をキャンセルしますが、先ほど部分的なキャンセルも可能だとおっしゃっていました。どちらが正しいのでしょうか？」

### CONTEXT.md を随時更新する

用語が解決されたら、その場で `CONTEXT.md` を更新してください。
まとめて更新するのではなく、発生した時点で記録してください。
[CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md) のフォーマットを使用してください。

`CONTEXT.md` には実装の詳細を一切含めないでください。
`CONTEXT.md` を仕様書、メモ帳、または実装決定事項のリポジトリとして扱わないでください。これは用語集であり、それ以外の何物でもありません。

### ADRは控えめに実施する

以下の3つの条件すべてを満たす場合にのみ、意思決定記録(ADR)を提案してください。

1. **取り消しが困難** ― 後から考えを変えるコストが大きい
2. **背景情報がないと意外** ― 後からそれを読む人が「なぜこのような方法を選んだのだろう？」と疑問に思う
3. **トレードオフの選択結果** ― 複数の選択肢があり、明確な理由に基づいて選択した

上記3つのうちいずれか1つでも欠けている場合は、ADRは不要です。
[ADR-FORMAT.md](./ADR-FORMAT.md)に記載されているフォーマットを使用してください。
