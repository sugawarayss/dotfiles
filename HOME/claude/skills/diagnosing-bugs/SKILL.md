# Diagnosing bugs

難易度の高いバグへの対処法。明確な理由がある場合にのみ、フェーズをスキップしてください。

コードベースを調査する際は、`CONTEXT.md`（存在する場合）を読んで関連モジュールの全体像を把握し、作業対象領域のADR（アプリケーション定義レポート）を確認してください。

## フェーズ1 - フィードバックループの構築

**これこそがスキルです。** それ以外はすべて機械的な作業です。
バグに対する**明確な**合否判定基準、つまり「このバグが発生したら赤信号になる」基準があれば、原因を突き止めることができます。
二分法、仮説検定、インストルメンテーションといった手法は、その基準を消費するだけです。
もし明確な基準がなければ、どれだけコードを見つめても無駄です。

ここには、他の要素を圧倒するほどの努力を注ぎ込んでください。**積極的に、創造的に、決して諦めずに取り組んでください。**

### 構築方法 ― おおよそこの順序で試してみてください

1. **テスト失敗**: ユニットテスト、統合テスト、エンドツーエンドテストなど、バグに到達するすべての段階で発生します。
2. **Curl / HTTPスクリプト**: 実行中の開発サーバーに対するリクエストの実施
3. **CLI呼び出し**: フィクスチャ入力を使用したCLI 呼び出しで、標準出力を既知の正常なスナップショットと比較します。
4. **ヘッドレスブラウザスクリプト** (Playwright / Puppeteer): UI を制御し、DOM/コンソール/ネットワークの内容を確認できます。
5. **キャプチャしたトレースを再生します。**: 実際のネットワークリクエスト/ペイロード/イベントログをディスクに保存し、コードパスを個別に再生します。
6. **使い捨てハーネス**: システムの最小限のサブセット（1つのサービス、モックされた依存関係）を起動し、単一の関数呼び出しでバグのコードパスを実行します。
7. **プロパティ/ファズループ**: バグが「時々間違った出力が発生する」場合、1000個のランダムな入力を実行し、障害モードを探します。
8. **バイセクションハーネス。**: バグが既知の 2 つの状態 (コミット、データセット、バージョン) の間に発生した場合、「状態 X で起動、チェック、繰り返し」を自動化して、`git bisect run` を実行できるようにします。
9. **差分ループ**: 同じ入力を旧バージョンと新バージョン（または2つの構成）に通し、出力の差分を取得します。
10. **HITL bashスクリプト**: 最終手段。どうしても人間がクリックする必要がある場合は、`scripts/hitl-loop.template.sh` を使って操作し、ループの構造を維持してください。キャプチャされた出力はフィードバックされます。

適切なフィードバックループを構築すれば、バグは90%修正できます。

### ループする範囲を狭める

ループをプロダクトとして扱います。ループができたら、それを**締め**ます。

- もっと高速化できる余地はないか？（キャッシュ設定、無関係な初期化処理のスキップ、テスト範囲の絞り込みなど。）
- バグの痕跡をより鮮明にする余地はないか？（「クラッシュしなかった」ではなく、具体的な症状について述べてください。）
- もっと決定論的にする余地はないか？（時刻の固定、乱数生成器のシード設定、ファイルシステムの隔離、ネットワークのフリーズなど）

長く不安定なループは、ループがないのと大差ない。
一方、短時間の確実なループは完璧で、デバッグにおける強力な武器となる。

### 非決定的なバグ

目標は完璧な再現手順ではなく、**再現率の向上**です。
トリガーを100回ループさせ、並列処理を行い、負荷を加え、タイミングウィンドウを狭め、スリープを挿入します。
50%の確率で発生するバグはデバッグ可能ですが、1%では不可能です。
デバッグ可能になるまで再現率を上げ続けましょう。

### 本当にループを構築できない場合

立ち止まって、その旨を明確に伝えましょう。
試したことを列挙してください。
ユーザーに以下のいずれかを求めてください。
(a) 問題を再現できる環境へのアクセス、
(b) 取得済みのアーティファクト（HARファイル、ログダンプ、コアダンプ、タイムスタンプ付き画面録画）
(c) 一時的な本番環境計測ツールの追加許可。

検証なしに仮説を立てることは**絶対に**しないでください。

### 完了条件：赤信号が点灯するまでループを絞る

フェーズ1は、ループが **タイト** かつ **赤信号** になった時点で完了です。
**既に少なくとも一度実行した** コマンドを **1つ** 指定してください（スクリプトのパス、テスト実行、curlなど）（実行したコマンドとその出力を貼り付けてください。）
つまり要点は以下です

- [] **赤信号**: 実際のバグコードパスを駆動し、**ユーザーの正確な症状** をアサートするため、このバグでは赤信号、修正後は青信号になります。「エラーなく実行される」だけでは不十分です。この特定のバグを **捕捉** できる必要があります。
- [] **決定論的**: 実行ごとに同じ判定（不安定なバグ、再現率が高い）。
- [] **高速**: 分でなく秒単位
- [] **エージェント実行可能**: AIエージェントが実行できるかどうか。人間の関与を必要とする場合は、`scripts/hitl-loop.template.sh` を使用してください。

このコマンドが存在する前に、理論を構築するためにコードを読み込んでいることに気づいたら、**すぐにやめてください。
仮説に飛びつくことは、まさにこのスキルが防ぐべき失敗です。** 赤信号検出がなければ、フェーズ2は進めません。

## フェーズ2 - 問題の再現と最小化

ループを実行してください。
赤信号が点灯することを確認してください。
バグが発生しています。

確認観点:

- [] このループは、**ユーザー** が説明した不具合を発生させるものであるか。(たまたま近くで発生している別の不具合ではありません。間違ったバグには間違った修正が適用されます。)
- [] この不具合は複数回の実行で再現可能であるか。（あるいは、非決定論的なバグの場合は、デバッグに十分な頻度で再現可能である)
- [] エラーメッセージ、誤った出力、処理速度の低下といった正確な症状を把握できたかどうか。後の段階で修正が実際に問題を解決していることを確認できます。

### 問題の最小化

赤信号が点灯したら、再現手順を **最小限のシナリオ** に縮小してください。
入力、呼び出し元、設定、データ、ステップを **一つずつ** 削除し、削除するたびにループを再実行してください。
不具合の原因となっている要素だけを残してください。

なぜわざわざそんなことをするのか: 最小限の再現手順を用いることで、フェーズ3における仮説の範囲が縮小し（疑わしい要素が少なくなる）、フェーズ5における明確な回帰分析となる。
**残りの要素すべてが荷重を支える状態**になった時点(いずれかの要素を取り除くと、ループが青信号に変わる状態)で完了です。

問題の再現と最小化が完了するまで先に進まないでください。

## フェーズ3 - 仮説をたてる

いずれかの仮説を検証する前に、**3～5個の順位付けされた仮説**を作成してください。
単一の仮説を作成する場合は、最初に妥当と思われるアイデアを基準とします。

それぞれの仮説は**反証可能**でなければならない。
つまり、その仮説がどのような予測を立てるかを明記しなければなりません。

> 形式：「もし<X>が原因であれば、<Yを変更する>とバグは解消される／<Zを変更する>とバグは悪化する。」

予測を明確に述べられないなら、その仮説は単なる予感に過ぎない。捨てるか、あるいはより明確なものに修正すべきだ。

**テスト前にユーザーにランキングリストを表示しましょう。**
ユーザーはドメイン知識を持っていることが多く、それによってランキングが即座に再調整されることがあります（「3位に変更を加えたばかりです」など）。
また、既に除外した仮説を把握している場合もあります。
これは手軽なチェックポイントであり、大幅な時間短縮につながります。
ユーザーが席を外している場合は、ランキング処理を中断せずに続行してください。

## フェーズ4 - 検証に利用するツール

各検証は、フェーズ3の特定の予測に対応していなければなりません。**一度に1つの変数のみを変更してください。**

検証手段の利用ガイドライン:

1. **デバッガー/REPLによる検査**(環境が対応している場合): ブレークポイント1つで、ログ10個分よりはるかに効率的です。
2. **対象のログ**: 仮説を区別する境界です。
3. 「すべてをログに記録してgrepする」なんてことは絶対にしない。

**すべてのデバッグログに**一意のプレフィックス（例：`[DEBUG-a4f2]`）をタグ付けしてください。最終的なクリーンアップは単一のgrepコマンドで済みます。タグ付けされていないログは残り、タグ付けされたログは削除されます。

**パフォーマンスブランチ** パフォーマンス低下の場合、ログは通常間違っています。代わりに、ベースライン測定（タイミングハーネス、`performance.now()`、プロファイラ、クエリプラン）を行い、その後、二分探索を実施してください。まず測定し、次に修正します。

## フェーズ5 - 修正の実施とリグレッションテスト

リグレッションテストは修正前に実施してください。ただし、適切なタイミングがある場合に限ります。

適切なシームとは、呼び出し箇所で発生する**実際のバグパターン**をテストで再現できるシームのことです。
利用可能なシームが浅すぎる場合（バグの発生には複数の呼び出し元が必要なのに、呼び出し元が1つのテストしか行っていない場合、あるいはバグをトリガーした一連の処理を再現できない単体テストなど）、リグレッションテストを行っても誤った確信を与えてしまいます。

**適切なシームが存在しない場合、それ自体が発見事項です。** この点を記録してください。
コードベースのアーキテクチャがバグの解決を妨げています。次のフェーズのためにこの件をフラグ付けしてください。

適切なシームが存在するならば:

1. 最小限に縮小した再現手順を、その継ぎ目で失敗するテストに変換してください。
2. そのテストが失敗することを確認します。
3. 修正を適用します。
4. テストが成功するようになることを確認します。
5. フェーズ1のフィードバックループを、元の（最小化されていない）シナリオに対して再実行します。

## フェーズ6 - 後始末と今後

完了を宣言する前に実施すること:

- [] 元の再現手順では再現しなくなることを確認（フェーズ1のループを再実行してください）。
- [] リグレッションテストに合格していることを確認（または継ぎ目がないことが確認されている）
- [] `[DEBUG-...]` のすべての計測タグが削除されていることを確認（プレフィックスを `grep` で検索してください）。
- [] 使い捨てハーネススクリプトが削除されることを確認（または明確に表示されたデバッグ場所に移動される）
- [] 正しいこと判明した仮説はコミット/プルリクエストメッセージに記載されていること。(次のデバッガーへのヒントになります)

**次に、「このバグを防ぐにはどうすればよいか？」と問いかけてください。**
答えがアーキテクチャの変更（適切なテストシームがない、呼び出し元が複雑に絡み合っている、隠れた結合があるなど）に関わる場合は、具体的な内容を`/improve-codebase-architecture`スキルに引き継いでください。修正が適用される**後**に推奨事項を提示してください。修正前ではありません。開始時よりも多くの情報が得られているからです。
