# レビュー観点

diff の各変更行を、以下の観点でチェックする。ここに挙げるのは正確性・バグに関わるものだけで、過剰設計や複雑性（complexity-review / audit-review の領分）、セキュリティ脆弱性（security-review の領分）は対象外とする。

## 1. ロジック・条件分岐

- 条件式の反転漏れ、比較演算子の誤り（境界値を含むか含まないか）
- ループの境界（off-by-one）
- 非同期処理・並行処理での順序保証や競合状態
- 分岐の網羅性（想定していない入力パターンが抜け落ちていないか）

## 2. エッジケース・入力値

- null / undefined / 空文字列 / 空配列 / 0 / 負数の考慮漏れ
- 型の不一致や暗黙変換に依存した処理
- 外部入力（API レスポンス、ユーザー入力、ファイル内容）の検証漏れ

## 3. エラー・例外処理

- 例外の握りつぶし（catch して何もしない、ログすら出さない）
- リソース解放漏れ（ファイルハンドル、コネクション、ロック）
- エラーメッセージ・ログ情報の不足（原因追跡できない）

## 4. 副作用・状態管理

- 関数名やシグネチャから予測できない副作用（例: getter なのに状態を変更する）
- 意図しない共有状態・グローバル状態の変更
- 呼び出し元が期待する範囲を超えた処理（PoLA 違反）

## 5. 既存コードとの整合性

- 命名規則や既存パターンからの逸脱
- 同一ファイル・同一モジュール内での書き方の不統一

## 6. テスト

- 新規追加・変更されたロジックに対応するテストが存在するか
- 既存テストが今回の変更で無意味になっていないか（モックが実装と乖離していないか等）

## 指摘しないこと

- 抽象化の是非、柔軟性の過不足、標準ライブラリで代替できるかどうか → complexity-review / audit-review に譲る
- インジェクション・認証・認可・秘密情報の扱いなど → security-review に譲る
- スタイル的な好み（フォーマットなど）で実害のないもの
