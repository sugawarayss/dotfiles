# Deepening

依存関係を考慮しながら、浅いモジュール群を安全に深層化する方法。[SKILL.md](SKILL.md)に記載されている用語（**モジュール**、**インターフェース**、**シーム**、**アダプタ**）を前提としています。

## 依存のカテゴリ

候補を深層化する際には、その依存関係をカテゴライズします。カテゴリは、深層化されたモジュールがシームを通じてどのようにテストされるかを決定します。

### 1. In-process

純粋な計算、メモリ内の状態、I/Oなし。常に深層化可能 — モジュールを統合し、新しいインターフェースを通じて直接テストします。アダプタは不要です。

### 2. Local-substitutable

ローカルテスト用の代替環境（PostgreSQLの場合はPGLite、インメモリファイルシステムなど）を持つ依存関係。代替環境が存在する場合は、詳細化が可能。詳細化されたモジュールは、テストスイート内で代替環境を実行してテストされます。このシームは内部的なものであり、モジュールの外部インターフェースにポートはありません。

### 3. Remote but owned (Ports & Adapters)

ネットワーク境界を越えたサービス（マイクロサービス、内部APIなど）。シームに**ポート**（インターフェース）を定義します。深層化されたモジュールがロジックを所有し、トランスポートは**アダプタ**として注入されます。テストではインメモリアダプタを使用し、本番ではHTTP/gRPC/キューアダプタを使用します。

推奨事項の形式：「シーム部分にポートを定義し、本番環境向けにHTTPアダプタ、テスト環境向けにインメモリアダプタを実装することで、ネットワークを介したデプロイであってもロジックを単一の深いモジュール内に収めることができます。」

### 4. True external (Mock)

サードパーティサービス（Stripe、Twilioなど）は、ユーザーが制御できません。deepeningモジュールは、外部依存関係を注入ポートとして受け取ります。テストでは、モックアダプタが提供されます。

## Seam discipline

- **アダプターが1つであれば、仮想的な継ぎ目を意味します。アダプターが2つであれば、実際の継ぎ目を意味します。** 少なくとも2つのアダプタが必要な場合を除き（通常は本番環境とテスト環境）、ポートを追加しないでください。アダプタが1つだけの接続は、単なる間接接続に過ぎません。
- **内部のシーム vs 外部のシーム** ディープモジュールは、インターフェース上の外部シームに加えて、内部シーム（実装固有のもので、自身のテストでのみ使用される）を持つことができます。テストで使用されるからといって、内部シームをインターフェースを通して公開しないでください。

## Testing strategy: replace, don't layer

- 浅いモジュールに対する古い単体テストは、より深いモジュールのインターフェースにテストが存在するようになった時点で無駄になるため、削除してください。
- 拡張されたモジュールのインターフェースに新しいテストを記述してください。**インターフェースはテストの表層です**
- テストは、内部状態ではなく、インターフェースを通じて観測可能な結果に基づいて検証を行う。
- テストは内部リファクタリング後も存続するべきである。なぜなら、テストは実装ではなく動作を記述するものだからだ。実装の変更に伴ってテストも変更しなければならないとしたら、それはインターフェースを超えたテストを行っていることになる。
