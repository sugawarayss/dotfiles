# Design It Twice

ユーザーが選択した深化候補に対して代替インターフェースを検討したい場合は、この並列サブエージェントパターンを使用します。「Design It Twice」（オスターハウト著）に基づいています。最初のアイデアが必ずしも最善とは限りません。

[SKILL.md](SKILL.md)の語彙を使用します — **モジュール**、**インターフェース**、**シーム**、**アダプター**、**レバレッジ**。

## 処理内容

### 1. 問題範囲のフレーミング

サブエージェントを生成する前に、選択された候補に対する問題範囲のユーザー向け説明を書きます:

- 新しいインターフェースが満たすべき制約
- それの依存関係と、それらがどのカテゴリに属するか ([DEEPENING.md](DEEPENING.md)を参照)
- 制約を具体的に示すための、大まかなコード例スケッチ。これは提案ではなく、制約を具体化するための方法である。

これをユーザーに表示したら、すぐにステップ2に進みます。ユーザーが読みながら考えている間、サブエージェントは並行して動作します。

### 2. サブエージェントの生成

Agentツールを使用して、3つ以上のサブエージェントを並行して生成します。各サブエージェントは、深められたモジュールに対して**根本的に異なる**インターフェースを生成する必要があります。

各サブエージェントに個別の技術概要（ファイルパス、結合の詳細、[DEEPENING.md](DEEPENING.md)からの依存関係カテゴリ、シームの背後にあるもの）を指示します。
この概要は、ステップ1のユーザー向け問題空間の説明とは独立しています。各エージェントに異なる設計制約を与えます。

- エージェント1: 「インターフェースを最小化 — エントリーポイントは最大で1〜3に抑える。各エントリーポイントのレバレッジを最大化。」
- エージェント2: 「柔軟性を最大限に高める ― 多様なユースケースと拡張性をサポートする。」
- エージェント3: 「モジュールの内部構造を反映するインターフェースを設計する — 変更がモジュール内にとどまるようにする。」
- エージェント4(適用可能であれば): 「シーム間の依存関係を考慮し、ポートとアダプタを中心に設計する。」

設計には、[SKILL.md](SKILL.md)の語彙とCONTEXT.mdの語彙の両方を含めてください。そうすることで、各サブエージェントがアーキテクチャ言語とプロジェクトのドメイン言語に沿って一貫性のある命名規則を使用するようになります。

それぞれのサブエージェントは以下を出力します:

1. インターフェース(型、メソッド、パラメータ — さらに不変条件、順序、エラーモード)
2. 呼び出し元がどのように使用するかを示す使用例
3. 実装がシームの背後に何を隠蔽するのか
4. 依存関係の戦略とアダプタ（[DEEPENING.md](DEEPENING.md)を参照）
5. トレードオフ — レバレッジが高い場所、薄い場所

### 3. 提示と比較

設計を順番に提示し、ユーザーが各設計を理解できるようにします。その後、文章で比較します。**深さ**（インターフェースでのレバレッジ）、**局所性**（変更が集中する場所）、**シームの配置**で対比します。

比較の後、自分の推奨を示します: どの設計が最も強力であると考えるか、そしてその理由を述べます。異なる設計の要素がうまく組み合わさる場合は、ハイブリッドを提案します。意見を明確に示してください — ユーザーは強い読み物を求めており、メニューを求めているわけではありません。
