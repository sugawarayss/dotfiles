# Codebase Design

**ディープモジュール**を設計しましょう。小さなインターフェースの背後に多くの動作を配置し、明確な境界線上に配置することで、そのインターフェースを通してテストできるようにします。コードの設計や再構築を行う際には、この考え方と原則を常に活用してください。目的は、呼び出し側にとっての利便性、保守担当者にとっての局所性、そしてすべての人にとってのテスト容易性を実現することです。

## 用語集

これらの用語は正確に使用してください。「コンポーネント」「サービス」「API」「境界」といった用語で代用しないでください。一貫性のある用語を使うことが何よりも重要です。

- **モジュール** - インターフェースと実装を持つものすべて。スケールに依存しないもの: 関数、クラス、パッケージ、または階層をまたぐスライス
  - **避けるべき用法** - ユニット、コンポーネント、サービス
- **インターフェース** - 呼び出し元がモジュールを正しく使用するために知っておくべきすべての情報。型シグネチャだけでなく、不変条件、順序制約、エラーモード、必要な設定、パフォーマンス特性なども含まれます。
  - **避けるべき用法**: API、シグネチャ（型レベルのインターフェースのみを指すため、範囲が狭すぎる）。
- **実装** - モジュールの中身、つまりコード本体のことです。**アダプタ**とは区別されます。例えば、小さなアダプタに大きな実装（PostgreSQLリポジトリなど）がある場合もあれば、大きなアダプタに大きな実装（メモリ上のフェイクなど）がある場合もあります。「アダプタ」は、その境界部分が話題になっている場合に使い、「実装」はそうでない場合に使います。
- **深さ** - インターフェースにおけるレバレッジ: 呼び出し元（またはテスト）が学習しなければならないインターフェースの単位あたりに実行できる動作の量。モジュールは、小さなインターフェースの背後に大量の動作が存在する場合、**深い**と言えます。インターフェースが実装とほぼ同じくらい複雑な場合、**浅い**と言えます。
- **シーム**（マイケル・フェザーズ）- その場所を編集せずに動作を変更できる場所。モジュールのインターフェースが存在する*場所*。シームをどこに配置するかは、その背後にあるものとは切り離された、独自の設計上の決定事項である。
  - **回避すべき用法**：境界（DDDの境界コンテキストと混同される）。
- **アダプター** - インターフェースの継ぎ目を満たす具体的なもの。中身（内部構造）ではなく、役割（どの役割を果たすか）を表す。
- **レバレッジ** - 呼び出し元が深い知識から得られるもの: 学習するインターフェース単位あたりの機能向上。1つの実装でN個の呼び出しサイトとM個のテストにわたって効果を発揮します。
- **局所性** - メンテナンス担当者が深層構造から得られるメリット: 変更、バグ、知識、検証が呼び出し元全体に分散するのではなく、一箇所に集中します。一度修正すれば、あらゆる場所で修正されます。

## 深さ vs 浅さ

**ディープモジュール** = 小さなインターフェース + 多数の実装:

```
┌────────────────────────────┐
│   小さいインターフェース   │  ← 数個のメソッド、シンプルなパラメータ
├────────────────────────────┤
│                            │
│         深い実装           │  ← 複雑なロジックが隠蔽される
│                            │
└────────────────────────────┘
```

**シャローモジュール** = 大きなインターフェース + 小さな実装（避けるべき）：

```
┌─────────────────────────────────┐
│    大きいインターフェース       │  ← 大量のメソッド、複雑なパラメータ
├─────────────────────────────────┤
│         薄い実行                │  ← ただ通りすぎるだけ
└─────────────────────────────────┘
```

インターフェースを設計する時は、こう自問しましょう。

- メソッドの数を減らすことはできないか?
- パラメータをシンプルにできないか?
- 複雑な内部をもっと隠蔽することはできないか?

## 信条

- **深さはインターフェースの特性であり、実装の特性ではありません。** 深いモジュールは、内部的に小さく、モック可能で、交換可能な部品で構成されている場合があります。ただし、それらはインターフェースの一部ではありません。モジュールは、インターフェースに**外部シーム**を持つだけでなく、**内部シーム**（実装に固有のもので、自身のテストで使用されます）も持つことができます。
- **削除テスト** モジュールを削除することを想像してみてください。複雑さがなくなる場合は、それは単なる通過処理でした。N 個の呼び出し元で複雑さが再び現れる場合は、そのモジュールは存在意義があったと言えます。
- *インターフェースはテスト対象の表層です。* 呼び出し元とテストは同じ境界を越えます。インターフェースを*超えて*テストしたい場合は、モジュールの形状が適切でない可能性があります。
- **アダプターが1つであれば、仮想的な継ぎ目となります。アダプターが2つであれば、実際の継ぎ目となります。** 継ぎ目に沿って実際に変化が生じない限り、継ぎ目を設けないでください。

## テスト可能性のための設計

良い設計が自然なテストを可能にします。

1. **依存を許容するが、依存を作らない**

   ```typescript
   // Testable
   function processOrder(order, paymentGateway) {}

   // Hard to test
   function processOrder(order) {
     const gateway = new StripeGateway();
   }
   ```

2. **副作用を発生させずに結果を返す**

   ```typescript
   // Testable
   function calculateDiscount(cart): Discount {}

   // Hard to test
   function applyDiscount(cart): void {
     cart.total -= discount;
   }
   ```

3. **小さな表層**

    少数のメソッドは少数のテストで済む。
    少数のパラメータはシンプルなテストで済む。

## 連携

- 1つの **モジュール** には確実に1つの **インターフェース** (呼び出し元やテストに対する表層)を持ちます。
- **深さ** とは **モジュール** が持つ属性です。 **インターフェース** の基準として観測されます。
- **シーム**とは、**モジュール**の**インターフェース**が存在する場所です。
- 1つの**アダプター** は 1つの**シーム** に位置し、インターフェースを満たします。
- **深さ**は呼び出し元に**レバレッジ**をもたらし、保守担当者に**局所性**をもたらします。

## 拒絶する枠組み

- **実装行数とインターフェース行数の比率としての深さ**（Ousterhout）：実装を詰め込むと有利になる。代わりに、深さをレバレッジとして使用する。
- **TypeScript の `interface` キーワードまたはクラスのパブリック メソッドとしての「インターフェース」**: 範囲が狭すぎます。ここでのインターフェースには、呼び出し元が知っておく必要のあるすべての事実が含まれます。
- **「境界」**: DDD の境界コンテキストで多義化されています。**シーム** や **インターフェース** など。
