# Improve Codebase Architecture

アーキテクチャ上の課題を表面化させ、**深化の機会**を提案する――浅いモジュールを深いモジュールへと変えるリファクタリングを行う。
目的はテスト容易性とAIによるナビゲーションの容易性である。

このコマンドは、プロジェクトのドメインモデルに基づいており、共通の設計語彙に基づいて構築されています。

- アーキテクチャ用語（**モジュール**、**インターフェース**、**深度**、**シーム**、**アダプター**、**レバレッジ**、**ローカリティ**）とその原則（削除テスト、「インターフェースはテスト面である」、「アダプターが1つ＝仮想的なシーム、2つ＝現実のシーム」）については、`/codebase-design` スキルを実行してください。提案の際には、これらの用語を正確に使用してください。「コンポーネント」「サービス」「API」「境界」といった曖昧な表現は避けてください。
- `CONTEXT.md` のドメイン言語は、適切なシームに名前を付けます。`docs/adr/` にある ADR は、このコマンドが再審理すべきでない決定を記録しています。

## 実施

### 1. 探索

まず、プロジェクトのドメイン用語集（`CONTEXT.md`）と、作業対象領域に関するすべてのADR（Advanced Document Review：関連文書）を読んでください。

次に、`subagent_type=Explore`を指定してエージェントツールを使用し、コードベースを探索してください。
厳密なヒューリスティックに従うのではなく、自然な流れで探索し、問題が発生する箇所を記録してください。

探索の観点:

- 一つの概念を理解するために、多くの小さなモジュールを行ったり来たりする必要があるのはどのような場合でしょうか？
- モジュールが**浅い**、つまりインターフェースが実装とほぼ同じくらい複雑なのはどこでしょうか？
- テスト容易性のためだけに純粋関数が抽出されているものの、実際のバグはそれらの呼び出し方法（**局所性**の欠如）に潜んでいる箇所はどこでしょうか？
- 密結合されたモジュールの、どの結合部から漏洩するのでしょうか？
- コードベースの中で、テストされていない部分、あるいは現在のインターフェースではテストが困難な部分はどれですか？

浅いと思われるものには、**削除テスト**を適用してください。削除することで複雑さが集中するのか、それとも単に移動するだけなのか？「集中する」という結果が、あなたが求めているシグナルです。

### 2. 改善箇所の候補をHTML形式のレポートとして出力

OSの一時ディレクトリに自己完結型のHTMLファイルを書き込み、リポジトリには何も書き込まれないようにします。
一時ディレクトリは`$TMPDIR`から取得し、解決できない場合は`/tmp`（Windowsの場合は`%TEMP%`）にフォールバックします。
そして、`<tmpdir>/architecture-review-<timestamp>.html`に書き込み、実行ごとに新しいファイルが生成されるようにします。
ユーザーがそのファイルを開くには、Linuxでは`xdg-open <path>`、macOSでは`open <path>`、Windowsでは`start <path>`を使用し、絶対パスをユーザーに伝えます。

このレポートでは、レイアウトとスタイリングに**CDN経由のTailwind**を、グラフ/フロー/シーケンスで構造が確実に伝わる図には**CDN経由のMermaid**を使用しています。Mermaidと手作業で作成したCSS/SVGビジュアルを組み合わせて使用してください。関係性がグラフ状になっている場合（呼び出しグラフ、依存関係、シーケンスなど）はMermaidを、より編集的な表現が必要な場合（マスダイアグラム、断面図、折りたたみアニメーションなど）は手作業で作成したdiv/SVGを使用してください。各候補には**変更前/変更後のビジュアライゼーション**が用意されています。視覚的に表現しましょう。

すべての改善候補は、カード形式で表示します。

カードに表示する情報:

- **ファイル**: 影響するファイルやモジュールの情報
- **問題**: なぜ現在のアーキテクチャが問題となるのか
- **改善方法**: なにを変更すべきなのかを説明する簡潔な日本語の文章
- **ベネフィット**: 影響範囲と影響度の観点から課題を説明し、テストがどのように改善されるか
- **変更前後の図**: 比較しやすいように並べて表示し、理解を深められるようにします。
- **修正のおすすめ度**: "強く推奨"、"修正価値有り"、"投機的" の何れかをバッジとして表示する

レポートの最後には、**最優先の推薦**セクションを設けてください。どの候補者を最初にアプローチすべきか、そしてその理由を述べてください。

**ドメインについてはCONTEXT.mdの用語を、アーキテクチャについては`/codebase-design`の用語を使用してください。**
`CONTEXT.md`で「Order」が定義されている場合は、「Order入力モジュール」と表現し、「FooBarHandler」や「Orderサービス」とは表現しないでください。

**ADRの競合**: 候補が既存のADRと矛盾する場合、その矛盾がADRの見直しを必要とするほど深刻な場合にのみ、その候補を取り上げてください。カードに明確に記載してください（例：「ADR-0007と矛盾するが、再検討する価値がある理由…」といった警告文）。ADRで禁止されているすべての理論的なリファクタリングを列挙しないでください。

HTMLの完全な構成、図表パターン、およびスタイルガイドについては、[HTML-REPORT.md](./HTML-REPORT.md)を参照してください。

現時点ではインターフェースを提案しないでください。ファイルを書き込んだ後、ユーザーに「これらのうちどれを試してみたいですか？」と尋ねてください。

### 3. 根掘り葉掘り

ユーザーが候補を選択したら、`/grilling` スキルを実行して、制約、依存関係、深層化されたモジュールの構造、継ぎ目の背後にあるもの、テストの残余など、設計ツリーをユーザーと共に確認します。

副作用は意思決定が具体化するにつれて発生するため、`/domain-modeling` スキルを実行してドメインモデルを常に最新の状態に保つようにしてください。

- **CONTEXT.md に記載されていない概念を、拡張モジュールの命名に使用したい場合** 該当の用語を `CONTEXT.md` に追加します。ファイルが存在しない場合は、遅延的に作成します。
- 会話中に曖昧な用語を明確にする必要がある場合は、その場で`CONTEXT.md`を更新してください。
- **ユーザーが負荷に影響を与える理由で候補を却下した場合？** 「今後のアーキテクチャレビューで同じ候補が再提案されないように、この件をADRとして記録しておきましょうか？」という形でADRを提案してください。理由が将来の担当者にとって実際に必要となる場合にのみ提案し、一時的な理由（「今は価値がない」など）や自明な理由は省略してください。
- **拡張モジュールの代替インターフェースを検討したいですか？** `/codebase-design` スキルを実行し、その design-it-twice 並列サブエージェントパターンを使用してください。
